## Overview

`__PACKAGE_NAME__` has feature parity with many other i18n libraries.
This means that you can use `__PACKAGE_NAME__` as a standalone i18n library, without using the General Translation platform.

To do this, don't provide any environment variables such as `GT_API_KEY` or `GT_PROJECT_ID`.

See our [migration guide](/docs/react/get-started/overview) for more information on how to migrate from another i18n library to `__PACKAGE_NAME__`.

## Tradeoffs

Using `__PACKAGE_NAME__` as a standalone i18n library has some tradeoffs.

### Manual translation

You will need to manually translate your app. If you use our platform, we automatically translate your app for you.

If your project only uses [dictionaries](/docs/react/guides/dictionaries) with the `useTranslations` function,
you'll need to manually translate your dictionaries, as you would with any other i18n library.

Make sure you load your translated dictionaries with the [`loadDictionary`](/docs/react/reference/functions/load-dictionary) function.

---

### Manual string translation

If your project is using inline translations with the [`<T>`](/docs/react/guides/translate-jsx) component
or the [`useGT`](/docs/react/guides/translate-strings) functions,
you'll also need to manually translate your strings.

Since there are no keys with inline translations, the CLI tool has a command: [`gt generate`](/docs/cli/reference/commands/generate)
which will automatically generate template files for your project. You'll just need to edit the template files with your translations for each language.

Make sure you load your translated strings with the [`loadTranslations`](/docs/react/reference/functions/load-translations) function.

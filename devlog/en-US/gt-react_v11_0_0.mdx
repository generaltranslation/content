## Overview

`gt-react` 11 is the center of a coordinated release across our React libraries. `gt-react`, `gt-next`, `@generaltranslation/react-core`, `gt-react-native`, and `gt-tanstack-start` all moved to version 11, alongside `gt-i18n` 1.0 and `generaltranslation` 9.0.

This release had two goals: make the i18n runtime easier to extend to more frameworks and reduce the amount of General Translation code shipped to users. The result is a shared foundation with clearer setup flows for browser-only and server-rendered apps.

## Updated setup flows

- **React SPAs** - Initialize `gt-react` with `initializeGTSPA()` before the app loads. The initializer resolves the locale and translations up front, so SPAs no longer need a provider. See the [React SPA Quickstart](/docs/react/react-spa-quickstart).
- **Server-rendered React** - Initialize the runtime once, then resolve the request locale and translation snapshot on the server. Pass both `locale` and `translations` to `<GTProvider>` for synchronous hydration. See the [server-rendered React Quickstart](/docs/react/react-quickstart).
- **Next.js Pages Router** - Use `withGTServerSideProps()` to add the locale and translation snapshot to page props, then hydrate `<GTProvider>` in `_app.tsx`. The v11 release line also adds middleware-based locale routing and static generation support through the [Pages Router middleware guide](/docs/react/nextjs/pages-router-middleware) and `withGTStaticProps()` in the [Pages Router static site generation guide](/docs/react/nextjs/pages-router-static-site-generation). Start with the [Pages Router Quickstart](/docs/react/nextjs-pages-router-quickstart).
- **TanStack Start** - The integration now uses the shared `gt-react` runtime instead of maintaining a separate provider stack. Resolve the locale and snapshot in the root loader, then pass both to `<GTProvider>`. Locale selection also persists through the configured locale cookie. See the [TanStack Start Quickstart](/docs/react/tanstack-start-quickstart).

`gt-react-native` now shares the same configuration and locale-storage foundation, keeping its behavior aligned with the rest of the React packages.

## A smaller, more extensible runtime

We split the browser-safe translation runtime from the full project and file-management API, made `@generaltranslation/react-core` tree-shakeable at the module level, and tightened package entry points so bundlers can remove unused code. Development-only translation paths can also be dropped from production builds.

In our production Next.js benchmark, GT-attributed client JavaScript fell from 251,086 bytes to 107,825 bytes: a **57.1% decrease**. This measures the General Translation portion of the client bundle, not the entire application bundle. Explore the package and entry-point data in the [Odysseus Bundle Size Tracker](https://odysseus-bundlesize.vercel.app/).

These changes also make framework integrations thinner: shared translation, configuration, and provider behavior lives in the common runtime, while each framework only supplies its environment-specific locale and request handling.

## Overview

`derive()` now works inside tagged template literals. Previously, `derive()` could only be used inside `t()` function calls. Now you can write:

```jsx
import { t } from "gt-react/browser";
import { derive } from "gt-react";

const label = t`Price: ${derive(price, { usd: "US Dollars", eur: "Euros" })}`;
```

The compiler recognizes `derive()` calls inside template expressions and preserves them as live expressions in the output, rather than collapsing them into `{n}` placeholders.

## What changed

### Compiler pipeline

The macro-expansion pass now uses a three-stage pipeline for processing template literals and string concatenation:

1. **`flattenExpressionToParts`** тАФ recursively decomposes each expression into static, dynamic, or derive parts
2. **`mergeAdjacentStaticParts`** тАФ collapses consecutive static parts into single strings
3. **`buildTransformResult`** тАФ converts the merged parts into the final AST node

This replaces the previous approach of assigning one `{n}` placeholder per expression. The new pipeline can distinguish between a regular variable (which becomes `{n}`) and a `derive()` call (which stays as a template expression).

### CLI scanner

The CLI's `handleDerivation` function now accepts a `runtimeInterpolationState` parameter. When scanning tagged template literals, non-derive dynamic values are assigned `{n}` placeholders, while `derive()` calls are fully resolved with their variant maps and UUIDs.

### Runtime

The `t` function gained a `handleTaggedTemplateLiteralTranslation` path. In development mode, it first tries looking up the fully-interpolated string (with derive values resolved) before falling back to the placeholder-based path. This ensures derive-resolved catalog keys work correctly during development.

## Before and after

```jsx
// Before тАФ derive only worked in t() calls
t("Price: {price}", { price: derive(amount, { usd: "US Dollars", eur: "Euros" }) })

// After тАФ derive works in tagged templates too
t`Price: ${derive(amount, { usd: "US Dollars", eur: "Euros" })}`
```

Both forms produce the same translation behavior. The tagged template version is more natural to write тАФ you don't need to name placeholders or pass an options object.

## Concatenation flattening

As part of this change, `+` binary expressions inside template literals are now recursively flattened. This means:

```jsx
t`Result: ${a + b}`
```

produces the catalog key `"Result: {0}{1}"` with two separate variables, rather than the previous `"Result: {0}"` with one combined variable. This is intentional тАФ it allows the compiler to detect `derive()` calls nested inside concatenation тАФ but it does change catalog keys for existing translations that use `+` inside template expressions.

## Overview

The `<GTProvider>` component provides General Translation (GT) context to its children in server-rendered React apps.
It receives the current `locale` and a `translations` snapshot, and renders synchronously — there is no loading state.

App-wide configuration does not live on the provider.
Configure `gt-react` once at startup with `initializeGT()` (locales, credentials, translation loaders); `<GTProvider>` only carries per-request state.

### When to use

- **Server-rendered apps**: call `initializeGT()` at startup, load a snapshot with `getTranslationsSnapshot(locale)` on the server, and pass it to `<GTProvider>` at the root of your component tree.

- **Single-page apps**: initialize with `initializeGTSPA()` instead. You don't need a `<GTProvider>` — components and hooks read from the global GT state.

### Description

| Prop            | Description                                                                                                                                          |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| `locale`        | The locale to render, e.g. `'fr'`. Typically resolved on the server from the request (cookie or `Accept-Language` header).                            |
| `translations`  | The translations snapshot for the current locale, e.g. the result of `getTranslationsSnapshot(locale)` fetched in a route loader or server handler.   |
| `children?`     | The subtree that needs access to translations. This should include any components using `<T>`, `useGT`, or other translation utilities.               |
| `dictionaries?` | A per-locale dictionary snapshot for [`useTranslations`](/docs/react/api/dictionary/use-translations).                                                |
| `region?`       | The user's region code (e.g., `US`, `GB`). Used for region-specific formatting.                                                                       |
| `enableI18n?`   | Whether to enable i18n features. Defaults to `true`.                                                                                                  |

### Returns

`React.JSX.Element` containing the children that were passed to this component.

## Examples

### Configure GT at startup

App-wide configuration lives in `initializeGT()`, not on the provider.
Call it once at module scope; the configuration is set once and immutable for the lifetime of the app.

```ts title="src/gt.ts" copy
import { initializeGT } from 'gt-react';
import gtConfig from '../gt.config.json';

initializeGT({
  ...gtConfig, // defaultLocale, locales, projectId, ...
  loadTranslations: async (locale: string) =>
    (await import(`./_gt/${locale}.json`)).default,
});
```

`initializeGT()` accepts your `gt.config.json` fields (`defaultLocale`, `locales`, `customMapping`, `projectId`, `devApiKey`, `apiKey`, `cacheUrl`, `runtimeUrl`) plus cache options (`loadTranslations`, `loadDictionary`, `dictionary`, `cacheExpiryTime`) and custom cookie names (`localeCookieName`, `regionCookieName`, `enableI18nCookieName`).

### Server-rendered apps

Fetch a translations snapshot on the server and pass it to `<GTProvider>` along with the resolved locale.
The provider renders synchronously from the snapshot.

```tsx title="src/root.tsx" copy
import { GTProvider, getTranslationsSnapshot } from 'gt-react';
import './gt'; // runs initializeGT()

// In your route loader (exact API depends on your framework)
export async function loader({ locale }: { locale: string }) {
  return {
    locale,
    translations: await getTranslationsSnapshot(locale), // [!code highlight]
  };
}

export default function Root({ locale, translations, children }) {
  return (
    <GTProvider locale={locale} translations={translations}> // [!code highlight]
      {children}
    </GTProvider>
  );
}
```

### Single-page apps

SPAs don't use `<GTProvider>`.
Call `initializeGTSPA()` in your entry point instead — it resolves once the initial translations are loaded, so render your app after it completes.

```tsx title="src/main.tsx" copy
import { createRoot } from 'react-dom/client';
import { initializeGTSPA } from 'gt-react';
import gtConfig from '../gt.config.json';
import App from './App';

await initializeGTSPA({
  ...gtConfig,
  loadTranslations: async (locale: string) =>
    (await import(`./_gt/${locale}.json`)).default,
});

createRoot(document.getElementById('root')!).render(<App />);
```

<Callout type="info">
  **Vite credentials:** During initialization, `gt-react` automatically reads `VITE_GT_PROJECT_ID` and `VITE_GT_DEV_API_KEY` from your environment. The dev API key is only read in development.
</Callout>

## Notes

* Configuration passed to `initializeGT()` or `initializeGTSPA()` is read once at startup and is immutable for the lifetime of the app.
* `<GTProvider>` renders synchronously from the `translations` snapshot — there is no loading state or fallback content.
* In server-rendered apps, the `<GTProvider>` must wrap all [`<T>` components](/docs/react/api/components/t) and translation hooks.
* In SPAs initialized with `initializeGTSPA()`, components and hooks work without a `<GTProvider>`.

`gt-next` adds automatic internationalization to Next.js App Router apps. A build-time plugin wires everything together, and the same [`<T>`](/docs/react/reference/components/t) component and [`useGT`](/docs/react/reference/hooks/use-gt) hook work in both server and client components.

Use this quickstart for Next.js apps using the App Router. For a plain React SPA use the [React Quickstart](/docs/react/react-quickstart).

*Note: `gt-next` supports the Next.js App Router. This quickstart assumes Next.js 13 or later.*

## Quickstart [#quickstart]

Install the packages, add the config plugin, create a config file, wrap your root layout, mark content for translation, and generate translations.

### 1. Install `gt-next`

Install `gt-next` as a dependency and the [`gt` CLI](/docs/cli/quickstart) as a dev dependency.

<Tabs items={['npm', 'yarn', 'bun', 'pnpm']}>
  <Tab value="npm">
    ```bash
    npm install gt-next && npm install gt --save-dev
    ```
  </Tab>

  <Tab value="yarn">
    ```bash
    yarn add gt-next && yarn add --dev gt
    ```
  </Tab>

  <Tab value="bun">
    ```bash
    bun add gt-next && bun add --dev gt
    ```
  </Tab>

  <Tab value="pnpm">
    ```bash
    pnpm add gt-next && pnpm add --save-dev gt
    ```
  </Tab>
</Tabs>

### 2. Add the config plugin

Wrap your Next.js config with `withGTConfig` from `gt-next/config`. The plugin reads your settings and sets up internationalization at build time — no other changes to your Next.js config are needed.

```ts title="next.config.ts"
import { withGTConfig } from 'gt-next/config';

const nextConfig = {};

export default withGTConfig(nextConfig);
```

### 3. Create `gt.config.json`

Create a `gt.config.json` file in your project root. It declares your source language and your target locales.

```json title="gt.config.json"
{
  "defaultLocale": "en",
  "locales": ["es", "fr", "ja"],
  "files": {
    "gt": {
      "output": "public/_gt/[locale].json"
    }
  }
}
```

- `defaultLocale` — the language your app is written in.
- `locales` — the languages to translate into. Pick from the [supported locales](/docs/platform/dashboard/reference/supported-locales).
- `files.gt.output` — where the CLI writes translation files. By default, `gt-next` loads translations from the General Translation CDN at runtime; see [Next.js configuration](/docs/react/nextjs/config) to serve them locally instead.

### 4. Add the provider to your root layout

Wrap your app in `GTProvider` at the root layout. In Next.js, `GTProvider` is a server component that loads translations on the server and passes them to your client components — you do not pass `locale` or `translations` props.

```tsx title="app/layout.tsx"
import { GTProvider } from 'gt-next';

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html>
      <body>
        <GTProvider>{children}</GTProvider>
      </body>
    </html>
  );
}
```

### 5. Mark content for translation

Wrap JSX in the `<T>` component to translate it in place. The same import from `gt-next` works in both server and client components. For standalone strings, use the `useGT` hook (client and server components) or the async [`getGT`](/docs/node/reference/functions/get-gt) from `gt-next/server` (server components only).

```tsx title="app/page.tsx"
import { T, useGT, LocaleSelector } from 'gt-next';

export default function Home() {
  const gt = useGT();

  return (
    <main>
      <LocaleSelector />
      <T>
        <h1>Welcome to my app</h1>
        <p>This content is translated automatically.</p>
      </T>
      <input
        placeholder={gt('Enter your email')}
        aria-label={gt('Email input field')}
      />
    </main>
  );
}
```

`useGT()` returns the translation function directly, so call it as `const gt = useGT();`. To translate strings in an async server component, use `getGT` instead:

```tsx title="app/page.tsx"
import { getGT } from 'gt-next/server';

export default async function Page() {
  const gt = await getGT();
  return <p>{gt('Hello, world!')}</p>;
}
```

### 6. Generate translations

Run the CLI's [`generate`](/docs/cli/reference/commands/generate) command to create your translation files. It scans your project for [`<T>`](/docs/react/reference/components/t) components and [`useGT`](/docs/react/reference/hooks/use-gt) calls and writes one editable file per locale to the `files.gt.output` path. No account or API key is required.

```bash
npx gt generate
```

Fill in the target-locale files by hand — standalone-string entries (from `useGT`) are stored as plain string values, while `<T>` JSX entries hold their content under a `c` field — or feed them to your own translation pipeline. Re-running `generate` picks up new content and preserves your edits.

To serve these files instead of the CDN, create a `loadTranslations.ts` file at your project root. The `withGTConfig` plugin detects it automatically.

```ts title="loadTranslations.ts"
export default async function loadTranslations(locale: string) {
  try {
    const t = await import(`./public/_gt/${locale}.json`);
    return t.default;
  } catch {
    return {};
  }
}
```

Restart the dev server and switch languages with the `LocaleSelector` — your translations render immediately. See [Storing translations locally](/docs/react/guides/storing-translations) for more on this setup.

### 7. Translate automatically

To machine-translate through the General Translation API instead of by hand, run [`translate`](/docs/cli/reference/commands/translate). It fills the same per-locale files, so the setup above keeps working.

```bash
npx gt translate
```

Add the command to your build script so production builds always have up-to-date translations:

```json title="package.json"
{
  "scripts": {
    "build": "npx gt translate && next build"
  }
}
```

*Note: `npx gt translate` needs a Project ID and a production API key, set as `GT_PROJECT_ID` and `GT_API_KEY` in your environment. Run `npx gt auth` or visit the [Dashboard](/docs/platform/dashboard/get-started) to get them. Never prefix these with `NEXT_PUBLIC_`.*

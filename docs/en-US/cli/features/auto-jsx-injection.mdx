## Overview

With auto JSX injection enabled, the CLI compiler automatically wraps translatable JSX text in `<T>` translation components at build time — so you don't need to manually add them around every piece of text.

Auto JSX injection is **disabled by default**.

<Callout type="warning">
Auto JSX injection currently only works with the new i18n-context system and for SPA React apps. Next.js server components are not yet supported.
</Callout>

## Enabling auto JSX injection

Set `enableAutoJsxInjection` to `true` in `files.gt.parsingFlags` in your `gt.config.json`:

```json title="gt.config.json"
{
  "files": {
    "gt": {
      "parsingFlags": {
        "enableAutoJsxInjection": true
      }
    }
  }
}
```

See the [configuration reference](/docs/cli/reference/config) for more on `parsingFlags`.

## How it works

Without auto JSX injection, you wrap translatable text manually:

```jsx
import { T } from 'gt-next';

function Welcome() {
  return (
    <T>
      <h1>Welcome to our app</h1>
    </T>
  );
}
```

With auto JSX injection enabled, you write plain JSX and the compiler handles the rest:

```jsx
function Welcome() {
  return <h1>Welcome to our app</h1>;
}
```

At build time, the compiler detects the translatable text and wraps it in `<T>` components automatically.

## What the compiler respects

- **User-written `<T>` components** are left untouched — the compiler won't double-wrap your code
- **Other GT components** like `<Var>`, `<Branch>`, `<Plural>`, and `<Derive>` are also respected
- **String attributes** (e.g., `placeholder`, `alt`) are not affected — only JSX children are wrapped
- **Non-text content** like numbers and booleans is ignored

## Limitations

The compiler processes files individually, so it cannot detect violations in re-exported components:

```jsx
// File A: export { T as Translate } from 'gt-next'
// File B: import { Translate } from './A'
<Translate>Hello {name}</Translate> // Won't be detected
```
